# Phase 4.1: Status Dashboard - Research

**Researched:** 2026-04-06
**Domain:** CLI dashboard — merging three data sources (worktrees, git branches, JIRA tickets) into grouped output
**Confidence:** HIGH

## Summary

Phase 4.1 adds a `treeji status` command that presents a unified dashboard across three data sources: git worktrees, all local branches, and JIRA assigned tickets. The core challenge is joining these three independent lists into seven logical states (combinations of which sources are present) and rendering them in two formats (compact default, `--full` detail).

The codebase already has all low-level primitives. The new code is pure orchestration: fetch all three sources in parallel, join them by ticket key, group by presence state, and render. No new dependencies are needed — chalk, execa, jira.js, and the existing helpers cover everything.

The design risk is performance: fetching JIRA tickets plus listing all branches plus listing worktrees must all happen concurrently. The existing pattern (Promise.all at the data-gather layer) handles this correctly and should be replicated verbatim.

**Primary recommendation:** Build `status.ts` as a pure orchestration layer over existing helpers. Add `gitListBranches()` to `git.ts`. Reuse `extractTicketKey()`, `colorStatus()`, `jiraCell()` pattern, and `fetchAssignedIssues()` without modification.

## Architecture Patterns

### Data Flow

```
treeji status
  ├── parallel fetch (Promise.all)
  │   ├── gitWorktreeList() → parseWorktreeList() → WorktreeInfo[]
  │   ├── gitListBranches() → string[]  (new helper)
  │   └── fetchAssignedIssues(jql?) → { key, summary, statusName }[]
  │
  ├── join phase
  │   ├── Build Map<ticketKey, JiraIssue> from JIRA results
  │   ├── Build Map<branchName, WorktreeInfo> from worktree list
  │   ├── Build Map<ticketKey, branchName> from branches (extractTicketKey)
  │   └── classify each item into one of 7 states → StatusEntry[]
  │
  └── render phase
      ├── compact (default): one-liner per entry
      └── --full: table format (columns like `list`)
```

### Pattern 1: `gitListBranches()` helper

Add to `src/lib/git.ts`. Follows the established pattern for all git helpers.

```typescript
// Pattern: execa + -C root + stdout parse (matches all existing git helpers)
export async function gitListBranches(): Promise<string[]> {
  const { stdout } = await execa('git', ['branch', '--list', '--format=%(refname:short)']);
  return stdout.trim() ? stdout.trim().split('\n') : [];
}
```

`--format=%(refname:short)` gives clean branch names without `* ` prefix or leading spaces — no regex trimming required. This is cleaner than `git branch --list` which requires stripping `* ` and `  ` prefixes.

### Pattern 2: Main/master filter (D-01)

Main/master branches are excluded by default unless they have local changes or unpushed commits. Since `gitStatusPorcelain` and `gitAheadBehind` require a worktree path (not a branch name), the check must be done against the main worktree path.

```typescript
// Get the main worktree (isMain === true from parseWorktreeList)
const mainWorktree = worktrees.find(wt => wt.isMain);
// Check if main should appear
let includeMain = false;
if (mainWorktree) {
  const [mainStatus, mainAB] = await Promise.all([
    gitStatusPorcelain(mainWorktree.path),
    gitAheadBehind(mainWorktree.path),
  ]);
  includeMain = mainStatus.trim().length > 0 || mainAB.ahead > 0;
}
// Filter branches list
const MAIN_BRANCHES = new Set(['main', 'master']);
const filteredBranches = branches.filter(b =>
  !MAIN_BRANCHES.has(b) || (MAIN_BRANCHES.has(b) && includeMain)
);
```

### Pattern 3: StatusEntry data structure

A single unified type that the classification step produces and the rendering step consumes:

```typescript
interface StatusEntry {
  state: 1 | 2 | 3 | 4 | 5 | 6 | 7;  // matches D-04 numbering
  jira: { key: string; summary: string; statusName: string } | null;
  branch: string | null;
  worktree: WorktreeInfo | null;
}
```

### Pattern 5: `--all` flag — JQL override

`fetchAssignedIssues` currently uses a fixed JQL. For `--all`, we need to drop the `statusCategory != Done` filter. Two approaches:

**Option A (recommended):** Add optional parameter to `fetchAssignedIssues`:
```typescript
export async function fetchAssignedIssues(
  maxResults = 50,
  includeAll = false
): Promise<...> {
  const jql = includeAll
    ? 'assignee = currentUser() ORDER BY updated DESC'
    : 'assignee = currentUser() AND statusCategory != Done ORDER BY updated DESC';
  ...
}
```

**Option B:** Pass raw JQL string as parameter. Option A is preferred — it keeps callers free of JQL knowledge.

This change to `jira.ts` is minimal and backward-compatible (default `false` preserves existing behavior used by `pick.ts`).

### Pattern 6: Compact format rendering

One line per entry. Arrow-separated segments, ANSI colors for status:

```
COREO-160 (In Progress) → feature/COREO-160-slug → ✓ worktree
COREO-180 (To Do)
feature/manual-work → ✓ worktree
feature/orphaned-branch
```

Section header style: chalk.bold + dim separator line (matches @clack/prompts aesthetic used in pick.ts).

### Pattern 7: `--full` flag rendering

Reuse the table-rendering approach from `list.ts` — padEnd columns, computed max-widths, OSC 8 hyperlinks via the `jiraCell()` pattern. The full format adds age and git status columns fetched via `gitStatusPorcelain` + `gitAheadBehind` for entries that have a worktree path.

For entries without a worktree (states 2, 3, 5), age and status columns show `—`.

### Anti-Patterns to Avoid

- **Sequential data fetching:** Do NOT await worktrees, then branches, then JIRA in sequence. Use `Promise.all([gitWorktreeList(), gitListBranches(), fetchAssignedIssues()])` — established pattern from `list.ts`.
- **Mutating jira.ts JQL string:** Do not hardcode a different JQL in `status.ts`. Extend `fetchAssignedIssues` with the `includeAll` parameter instead.
- **Duplicating `colorStatus` or `extractTicketKey`:** Both are already exported from `list.ts`. Import them — don't copy.
- **Calling `gitStatusPorcelain` for branches that lack a worktree:** These calls would fail (no directory to run `-C` against). Only call per-worktree git helpers when `entry.worktree !== null`.
- **Forgetting main worktree in branch dedup:** The main worktree appears in `parseWorktreeList` with `isMain: true` and a branch (e.g., `main`). It also appears in `gitListBranches()`. Without filtering, `main` gets double-counted.

## Common Pitfalls

### Pitfall 1: Double-counting main branch
**What goes wrong:** `main` appears in both `parseWorktreeList()` (as the main worktree's branch) and `gitListBranches()`. Without deduplication or filtering, main ends up in two sections.
**Why it happens:** The two data sources are independent.
**How to avoid:** Apply the D-01 filter (exclude main/master unless dirty/unpushed) before classification. Since main is typically the main worktree, it will be excluded from the branch list while still appearing through the worktree data source if needed.
**Warning signs:** Main worktree shows up twice in output.

### Pitfall 2: JIRA fetch failure blocks entire command
**What goes wrong:** If JIRA is unreachable, throwing unhandled rejects the entire `Promise.all`, producing no output at all.
**Why it happens:** `Promise.all` fails fast on any rejection.
**How to avoid:** Wrap the JIRA fetch in try/catch inside `Promise.all` using `Promise.allSettled` or catch inside the async function:
```typescript
let jiraIssues: JiraIssue[] = [];
let jiraWarning = false;
try {
  jiraIssues = await fetchAssignedIssues(...);
} catch {
  jiraWarning = true;
}
```
Run git operations (which never need network) unconditionally. Show JIRA-less view with warning. Matches the degradation pattern in `list.ts`.

### Pitfall 3: `gitStatusPorcelain` called on non-existent worktree path
**What goes wrong:** For state 2 (JIRA + Branch, no Worktree) and state 5 (orphaned branch), there is no worktree directory. Calling `gitStatusPorcelain(undefined)` or with an invalid path throws an execa error.
**Why it happens:** The `--full` format wants status for all entries.
**How to avoid:** Always guard: `entry.worktree !== null` before calling per-worktree git helpers. Render `—` for status/age/ahead/behind when no worktree.

### Pitfall 4: State 7 (JIRA + Worktree, no Branch) misclassified
**What goes wrong:** Worktree with a branch that doesn't match any JIRA ticket key gets placed in state 4 (Branch + Worktree, no JIRA) instead of state 7 when the worktree's branch was deleted but the worktree still exists.
**Why it happens:** Porcelain output always has `branch` field if a branch is checked out. State 7 is specifically for when `extractTicketKey(worktree.branch)` finds a key that matches a JIRA ticket, but the branch doesn't appear in `gitListBranches()` (e.g., branch was deleted).
**How to avoid:** State 7 detection: `worktree.branch` extracts a ticket key AND that key is in `jiraByKey` AND the branch is NOT in `filteredBranches`. This is rare but must be handled per D-04.

### Pitfall 5: `--all` flag exposing Done tickets that also have matched branches
**What goes wrong:** With `--all`, a Done ticket whose branch still exists locally gets shown as state 1 (Active work), misleading the user.
**Why it happens:** Classification is agnostic to status category.
**How to avoid:** This is correct behavior per D-02 — `--all` shows everything including closed. No special handling needed. The ticket's status (e.g., "Done" in green) makes the state clear. Document as intentional.

### Pitfall 6: `git branch --list` includes `HEAD` in detached HEAD state
**What goes wrong:** In detached HEAD state, `git branch --list` may output `* (HEAD detached at abc1234)` which doesn't parse cleanly.
**Why it happens:** `--list` without `--format` includes the `* ` prefix and special detached output.
**How to avoid:** Use `--format=%(refname:short)` which skips detached HEAD entirely (returns empty string for that entry). Filter empty strings from the result array.

```typescript
// Source: git-scm.com/docs/git-branch (--format option)
export async function gitListBranches(): Promise<string[]> {
  const { stdout } = await execa('git', ['branch', '--list', '--format=%(refname:short)']);
  if (!stdout.trim()) return [];
  return stdout.trim().split('\n').filter(Boolean);
}
```

### Section rendering pattern

```typescript
function renderSection(title: string, entries: StatusEntry[], full: boolean, host?: string): void {
  if (entries.length === 0) return; // D-08: hide empty sections
  console.log(chalk.bold(title));
  for (const entry of entries) {
    if (full) renderFullRow(entry, host);
    else renderCompactRow(entry, host);
  }
  console.log(); // blank line between sections
}
```

### Compact row for state 1

```typescript
// COREO-160 (In Progress) → feature/COREO-160-slug → ✓ worktree
function renderCompactRow(entry: StatusEntry, host?: string): void {
  const parts: string[] = [];
  if (entry.jira) {
    const url = host ? `${host}/browse/${entry.jira.key}` : '';
    const link = `\x1b]8;;${url}\x1b\\${entry.jira.key}\x1b]8;;\x1b\\`;
    parts.push(`${link} (${colorStatus(entry.jira.statusName)})`);
  }
  if (entry.branch) parts.push(chalk.cyan(entry.branch));
  if (entry.worktree) parts.push(chalk.green('✓ worktree'));
  console.log('  ' + parts.join(chalk.dim(' → ')));
}
```
